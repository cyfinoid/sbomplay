About SBOM Play

Learn more about this client-side SBOM analysis tool

Project Overview

SBOM Play is a client-side web application for analyzing Software Bill of Materials (SBOM) data from GitHub repositories, organizations, and users. Built for security professionals to identify dependency vulnerabilities, assess license compliance, and understand software supply chain risks in real-time.

The tool features comprehensive SBOM analysis including dependency tracking, vulnerability detection via OSV.dev integration, license compliance checking, author analysis with funding detection, and SBOM quality assessment.

Key Principle: All analysis happens directly in your browser - no data ever leaves your machine.

Features
  • SBOM Analysis: Analyze SBOM data from GitHub organizations, users, and repositories
  • Dependency Tracking: Track dependency usage across multiple repositories
  • Vulnerability Detection: OSV.dev integration for vulnerability scanning
  • License Compliance: Comprehensive license categorization and risk assessment
  • Author Analysis: Author deduplication and funding opportunity detection
  • SBOM Quality Assessment: Quality scoring based on multiple categories
  • Multi-Organization Storage: Persistent storage using IndexedDB
  • Export/Import: Export and import analysis data with checksum validation
Privacy & Security
Privacy Assured: All analysis happens in your browser. No data is sent to any server.

SBOM Play is designed with privacy and security as top priorities:

  • Client-Side Processing: All SBOM analysis, dependency resolution, and vulnerability checking happens entirely in your browser
  • No Data Transmission: Your SBOM data never leaves your machine - it's processed locally
  • Local Storage: Analysis results are stored in your browser's IndexedDB, giving you full control
  • API Calls: Only public registry APIs are queried (npm, PyPI, crates.io, etc.) - no sensitive data is transmitted
  • GitHub Token: Optional GitHub Personal Access Token is used only for API rate limits and is never stored

How It Works
  1. Input: Enter a GitHub organization name, username, repository, or GitHub URL
  2. SBOM Fetching: The tool queries GitHub's Dependency Graph API to retrieve SBOM data for public repositories
  3. Dependency Resolution: Full dependency trees are resolved by querying package registries (npm, PyPI, crates.io, etc.)
  4. Analysis: Dependencies are analyzed for vulnerabilities (OSV.dev), licenses, authors, and quality metrics
  5. Storage: Results are stored locally in your browser's IndexedDB for future reference
  6. Visualization: View results across multiple pages: Overview, Licenses, Vulnerabilities, Quality, Dependencies, and Authors
End-of-Life (EOX) Detection Methodology

SBOM Play uses multiple data sources and heuristics to identify end-of-life (EOL) and end-of-support (EOS) dependencies. Understanding which software components are no longer maintained is critical for supply chain security.

1. Confirmed EOL/EOS (Primary Source)

We query the endoflife.date API for official end-of-life information. This database tracks EOL dates for major software products including:

  • Programming languages: Python, Node.js, Ruby, PHP, Go, Java, .NET, Rust
  • Frameworks: Django, Rails, Laravel, Angular, React, Vue, Spring
  • Databases: PostgreSQL, MySQL, MongoDB, Redis, Elasticsearch
  • Infrastructure: Kubernetes, Docker, Terraform, Nginx, Apache
Definitions:
• EOL End-of-Life: No updates of any kind (features, bug fixes, or security patches)
• EOS End-of-Support: No security updates, but may still receive critical bug fixes
2. Probable EOL (Staleness Heuristic)

When official EOL data is unavailable, we use staleness analysis to infer probable abandonment. Research shows that packages without updates for extended periods are likely unmaintained.

ThresholdClassificationRationale
24+ months (2 years) Probable EOL Most active packages release at least once per year. 2 years of inactivity suggests abandonment.
36+ months (3 years) Highly Likely EOL Very high confidence of abandonment. Security vulnerabilities likely unpatched.

Note: This heuristic has limitations. Some stable packages (e.g., small utilities) may be "complete" and not require updates. We recommend manual review for probable EOL findings.

3. Archived Repositories

When a GitHub repository is archived, it's a clear signal from the maintainer that the project is no longer active. We detect dependencies whose source repositories have been archived and flag them accordingly.

Archived = Definitive Signal: Unlike staleness (which is probabilistic), an archived repository is an explicit declaration by the maintainer that development has ceased.
References
  • endoflife.date API Documentation
  • OpenEOX Standard - Emerging standard for EOL data exchange
  • GitHub: Archiving Repositories
Dependency Confusion Detection Methodology

Dependency confusion (also known as "namespace confusion" or "substitution attacks") is a supply chain attack where an attacker publishes a malicious package to a public registry with the same name as an internal/private package. When developers or CI/CD systems resolve dependencies, they may inadvertently download the malicious public package instead of the intended private one.

Real-World Impact: This attack vector has been used successfully against major organizations including Apple, Microsoft, PayPal, Shopify, Netflix, and many others, as disclosed by security researcher Alex Birsan in 2021.
How SBOM Play Detects Dependency Confusion

SBOM Play uses a multi-layered approach to detect potential dependency confusion vulnerabilities in your SBOMs:

Detection TypeSeverityDescription
Namespace Not Found HIGH The namespace/organization (e.g., @mycompany/ in npm or com.mycompany in Maven) doesn't exist in the public registry. An attacker could register the entire namespace and publish malicious packages. This is the highest-confidence indicator.
Package Not Found MEDIUM The specific package doesn't exist in the public registry. Could be a private/internal package that an attacker could claim by publishing a package with that name.
GitHub Action Missing HIGH For GitHub Actions, the owner/organization doesn't exist on GitHub. An attacker could create the organization and publish a malicious action.

Detection Process
  1. PURL Parsing: Extract ecosystem and package name from Package URLs (PURLs) in the SBOM
  2. Namespace Extraction: Parse scoped packages (e.g., @scope/package) to identify namespaces
  3. Registry Query: Check if namespace/package exists via ecosyste.ms or direct registry APIs
  4. GitHub Verification: For GitHub Actions, verify organization and repository exist via GitHub API
  5. Evidence Collection: Store API response URLs as proof for findings
Limitations:
  • Detection is based on public registry lookups - private registries are not checked
  • Some packages may be intentionally unpublished (deprecated) rather than private
  • Rate limits on registry APIs may affect detection completeness
  • False positives may occur for system packages (e.g., Python built-ins)
Mitigation Strategies
  • Namespace Squatting: Register your organization's namespace on public registries (even if unused)
  • Scoped Packages: Use organization scopes (e.g., @mycompany/) for all internal packages
  • Private Registry Priority: Configure package managers to check private registries first
  • Package Lock Files: Use lock files with integrity hashes to prevent substitution
  • Registry Allowlisting: Configure CI/CD to only allow packages from approved registries

Beyond the staleness heuristic covered in the EOX section above, SBOM Play also reads authoritative lifecycle signals directly from each ecosystem's native registry. When a package (or its publisher) has explicitly marked a release as deprecated, yanked, archived, or quarantined, that signal is treated as ground truth and surfaced separately from the heuristic so reviewers can tell first-party declarations from inferences.

Replacement-Hint Extraction

When a deprecation message names a successor (npm in particular often writes free-text deprecation strings), SBOM Play extracts the replacement package id with the following patterns and surfaces it in the package-details modal:

  • use X instead, migrate to X, switch to X, moved to X
  • superseded by X
  • NuGet's structured alternatePackage.id field
Limitations:
  • Lifecycle results are cached for 7 days per package to avoid hammering registries.
  • "Latest tag deprecated" on npm is a softer signal than "this exact version deprecated" and is labelled accordingly.
  • Ecosystems without a native deprecation channel rely on the GitHub archived fallback, which only fires when sourceRepoUrl resolves to a GitHub repository.
  • Some packages are intentionally feature-complete and never updated — manual review is recommended for borderline cases.

Malicious Package Detection Methodology

Malicious packages — intentionally hostile code published under typosquatted names, hijacked maintainer accounts, or hidden behind dependency confusion — are a different class of supply-chain risk from accidental CVEs in legitimate packages. SBOM Play splits these into their own Malware page and Findings category so the headline vulnerability counts elsewhere don't conflate "this library has a CVE" with "this library is the attack".

Treated as Critical: Every malware finding is assigned Critical severity by the Findings page — there is no "low-severity malicious package".
Detection Source

Malware advisories are pulled from OSV.dev, which mirrors the OpenSSF Malicious Packages dataset alongside regular CVE/GHSA advisories. The same OSV query that drives the Vulnerabilities page also surfaces MAL-* entries; SBOM Play then routes those into a separate malwareAnalysis bucket.

Classifier Rules

An OSV advisory is classified as malware if any of the following hold:

SignalField
Explicit kind fieldvuln.kind === 'malware'
Advisory id prefixid starts with MAL-
OpenSSF origin metadataaffected[].database_specific['malicious-packages-origins'] present
Database-specific flagdatabase_specific.malicious === true
False-Positive Mitigation: introduced: 0 Filter

The OpenSSF Malicious Packages dataset frequently emits an affected[].ranges[] entry of { introduced: "0" } with no matching fixed or last_affected bound — effectively "all versions ever published". When the same advisory also includes a specific versions[] list, the trivial range is treated as a placeholder and ignored. Without this filter, a malicious-version advisory would over-flag every other version of the same package (including legitimate predecessors).

Version Applicability

For each candidate advisory, advisoryAppliesToVersion matches the dependency's version against the OSV affected[] disjunction:

  • If versions[] is non-empty — exact-match required.
  • If only ranges[] is present — SemVer-style comparison (excluding the trivial introduced: 0 case above).
  • If affected[] is empty — conservatively kept as a match unless the cache has hydrated a more specific rule.
  • If the dependency version is unknown — advisory is kept (the user is shown an "unverified version" caveat).
Why Malware Is Excluded From Vulnerability Counters

The canonical OSVService.countUniqueAdvisories(...) helper used by every "Open Critical+High" tile, severity row, and headline number explicitly skips kind === 'malware' and MAL-* ids (along with withdrawn and VEX-suppressed advisories). This is what keeps the headline total equal to Critical + High + Medium + Low on the Vulnerabilities page.

Limitations:
  • The full OpenSSF index is not fetched live in-browser; only OSV-mirrored advisories are seen by default.
  • Conservative match for empty affected[] can produce false positives until the OSV cache hydrates a stricter rule.
  • Version comparison is pragmatic SemVer-style and not a complete ecosystem-specific resolver (e.g. PyPI PEP 440 epochs are best-effort).
References
  • OpenSSF Malicious Packages dataset
  • OSV Schema documentation
  • OSV.dev vulnerability database
VEX / VDR Consumption Methodology

VEX (Vulnerability Exploitability eXchange) and VDR (Vulnerability Disclosure Report) documents let downstream operators record their own exploitability decisions: "we ship CVE-YYYY-NNNN, but we're not affected because the vulnerable code path isn't reachable in our build." SBOM Play accepts uploaded VEX documents and uses them to annotate existing OSV findings with status, justification, and provenance.

Annotate, never delete: A VEX statement never causes an underlying CVE row to disappear from storage. The vulnerability remains in the dataset; the VEX label tells the user what the operator decided about it. Hiding suppressed findings in the UI is opt-in via a toggle.
Supported Formats
FormatDetection HeuristicNotes
CycloneDX VEXbomFormat === 'CycloneDX' with a vulnerabilities[] blockMost common format; covers full CycloneDX VDR with analysis blocks
OpenVEX@context contains openvex, or top-level statements[] with a vulnerability fieldLightweight, statement-oriented format
CSAF 2.0document.csaf_version or product_tree patternsJSON only — XML CSAF is rejected in-browser
Canonical Status Values

Every parsed statement is normalised into one of four canonical statuses (plus an unknown fallback):

affected — The product is exploitable via this vulnerability.
not_affected — The vulnerable code is present but not exploitable in this build / configuration.
fixed — The vulnerability has been remediated in this product version.
under_investigation — Triage is still in progress; status will be updated.

CycloneDX vendor-specific values are mapped automatically: resolved / resolved_with_pedigree → fixed, false_positive → not_affected, in_triage → under_investigation, exploitable → affected.

Matching Precedence

Each VEX statement is matched against the analysis's vulnerable dependencies using the following identifier order — the first match wins, preventing weaker signals (PURL aliasing, hash collisions) from over-applying:

  1. bom-ref — If both the SBOM and the VEX statement reference the same internal id, that's an exact pairing.
  2. purl — PURL match is lenient on the SBOM side: type and name must match; namespace and version are compared only when present on both sides; PURL qualifiers (?type=...) are ignored.
  3. Hash — Algorithm name is case-insensitive; hash value is compared strictly.

Vulnerability ids are matched against the union of vuln.id and vuln.aliases[], so a CSAF document referencing a CVE id will correctly attach to an OSV finding whose canonical id is the corresponding GHSA.

UI Suppression Rules

On the Vulnerabilities page, an opt-in toggle hides only findings whose VEX status is not_affected or fixed. Findings labelled affected or under_investigation remain visible regardless of the toggle — both still demand attention. Statements that don't match any dependency surface on the Findings page so the user can spot broken PURLs or typo'd CVE ids without inspecting each upload manually.

Limitations:
  • If multiple statements match the same finding, the most recently parsed one wins on vuln.vex; full history is preserved under vex.history.
  • Conflicting statements are not arbitrated automatically beyond last-write-wins.
  • XML CSAF documents are not supported; convert to JSON CSAF first.
  • The unknown fallback status is never treated as authoritative suppression.

Direct vs Transitive Classification & Dependency Depth

A direct dependency can be moved by a single line in the consumer repository's manifest (package.json, requirements.txt, pom.xml, etc.). A transitive dependency requires the maintainer of the depending package to ship a new version first. Treating both the same way overstates how actionable a finding is, so SBOM Play classifies every dependency on a per-(dep, repo) basis.

Per-pair classification: The same name@version can be direct in repository A (declared in its manifest) and transitive in repository B (pulled in by some other dep). Classification is always relative to the consuming repo, never global.
Ground-Truth Source

repo.directDependencies is populated only at SBOM parse time from the actual manifest entries the SBOM advertises — it is never overwritten by the registry resolver that builds out the transitive tree. The buildDirectMap helper produces a repoKey → Set(direct dep keys) lookup that every aggregator, table, and chart consults to assign the Direct or Transitive badge consistently.

Depth Computation (BFS)

For each repository, breadth-first search starts from the keys in directDependencies (Level 1) and walks outward through the inverted parents[] edges (built into a childrenByParent map). When multiple paths reach the same dependency, the shortest one wins.

Level-1 Imputation Rule

Real-world SBOMs — especially those produced by GitHub's Dependency Graph — routinely omit the dependsOn / DEPENDS_ON edges that BFS needs. When a dependency is present in repo.dependencies but unreachable via any parent chain (because the SBOM has no edges, the version mismatched the resolver-discovered parent, or the resolver hit maxDepth), it's bucketed at Level 1 — treated as direct — and counted under an imputedDirectGlobal tally.

The Insights "Portfolio-wide depth distribution" chart surfaces this count under the bars as "Includes N dependency occurrences with no traceable parent path; treated as direct", so the count is never silently hidden. (The previous "Unknown" bucket was removed because it conflated unreachable-real-deps with cross-repo spillover from an old resolver bug.)

Where The Split Surfaces

The Direct / Transitive split is visible across every finding view, so reviewers always know which side of the tree they're looking at:

  • Vulnerabilities page — "Filter by Reach" dropdown plus per-row "Direct in N / Transitive in M" badges; severity tiles show split values.
  • Licenses page — same Reach filter, plus per-card direct/transitive sub-counts on risk cards.
  • Insights page — dedicated "Vulnerabilities × dependency depth" stacked chart, "Direct-dep CVE dwell" KPI tile, and per-repo Critical+High mini-bars rendered as 4-segment stacks.
  • Findings, Malware, Audit, Repos pages — reach badges next to each affected package.
  • Per-repo CSV export — ships critical_direct, critical_transitive, high_direct, high_transitive, major_drift_direct, major_drift_transitive, direct_deps, and transitive_deps columns.
Limitations:
  • SBOMs without dependsOn edges produce many Level-1-imputed deps; the imputation count is the honest signal, not the precise tree.
  • Insights uses repo.dependencies as the per-repo scope (not dep.repositories) to avoid cross-repo spillover.
  • Resolver-discovered transitives that no SBOM listed (registry-only orphans) end up with empty repositories and correctly stop appearing in per-repo aggregations.
Tech-Debt Composite Scoring (Insights)

The Insights page produces a single portfolio-level health score on a 0–100 scale (higher = healthier) plus an A–F letter grade, designed for engineering managers, CTOs, and M&A reviewers who need a one-glance answer to "how much accumulated supply-chain debt does this codebase carry?"

Vulnerability Density Formula

Severities are weighted, with direct-dep findings counted at 3× the transitive rate:

weighted   = (critDirect * 30 + highDirect * 12 + medDirect * 3)
           + (critTrans  * 10 + highTrans  *  4 + medTrans  * 1)
vulnScore  = clamp01(weighted / max(1, totalDeps))
Drift Formula
majorWeighted = (major.direct * 3) + major.transitive
minorWeighted = (minor.direct * 3) + minor.transitive
driftScore    = clamp01(driftMajorRatio * 0.75 + (driftMinorRatio * 0.25 / 3))
Why 3× for direct deps? A direct-dep CVE is fixable today with a single bump in the consuming repo's manifest. A transitive-dep CVE typically requires the depending package's maintainer to release first. Two portfolios with identical headline counts can have very different actionability profiles — the 3× multiplier surfaces that.

  • Vulnerability age stats skip kind === 'malware'; malware presence still bumps the Hygiene component as a binary signal.
  • Direct/transitive splits use the SBOM-truth directMap, so portfolios with poor SBOM edge fidelity will see imputed-direct skew.
  • The composite is opinionated — weights are tuned for a "supply-chain reviewer" perspective, not a generic "code quality" view.

It is intentionally narrower than the per-page detail views — it surfaces discrete actionable findings, not raw advisory rows.

Severity Bucketing

The summary tiles at the top of the page bucket every finding instance:

  • Total — raw row count (every (finding, repo) pair).
  • Critical — severity === 'critical' (malware always lands here).
  • High — severity === 'high' or 'error'.
  • Medium — severity === 'medium'.
  • Warning — severity === 'warning'.

Per-category tiles in the second row sum instances, so the Total tile equals the sum of the visible per-category tiles by construction. Type groups within each category are sorted by severity order, then by instance count.

Direct / Transitive Reach Badges

Every applicable row picks up a Direct or Transitive badge derived from the same directMap the rest of the app uses, so a package can correctly show as Direct in one repo's row and Transitive in another repo's row of the same category.

OPML Feed Export Methodology

The Feeds page resolves a release-notification feed URL for every dependency in your portfolio, then exports the lot as an OPML 2.0 outline you can drop into any RSS/Atom reader (Feedly, NetNewsWire, FreshRSS, Inoreader, etc.). This lets you find out about new releases — including the security-fix releases you actually care about — without polling registries by hand.

URL Resolution Chain

_extractGitHubRepoFromDep walks the following sources in order to locate the repo that backs each dependency. The first hit wins; later sources are ignored.

  1. SBOM externalRefs — vcs, repository, website, documentation, plus URLs embedded in the purl qualifiers.
  2. dep.repositoryUrl / dep.raw.repositoryUrl — whatever the SBOM parser captured directly.
  3. dep.sourceRepoUrl — populated by the deps.dev enrichment phase from links[].label === 'SOURCE_REPO' / 'ORIGIN' / 'HOMEPAGE'.
  4. cacheManager.getPackageSync(ecosystem:name).repositoryUrl — whatever the native registry record (npm repository.url, PyPI source_code_uri, Cargo / RubyGems repository.url) had stored on a previous run.
deps.dev Backfill For Legacy Analyses

Older analyses captured before the sourceRepoUrl field existed will frequently come up as Uncovered. The Feeds page exposes a "Resolve missing source repos" button that issues a deduped batch of deps.dev links[] lookups (queued through the shared rate-limit lanes), persists the results to the package cache, and re-runs feed resolution in place. After a single backfill pass, the GitHub Releases fallback typically lights up for the majority of npm / Maven / NuGet / Cargo / Hex / Pub / CocoaPods packages whose registry record has a repository.url but whose SBOM didn't include an externalRef.

OPML 2.0 Output Shape

The exported OPML is grouped by reach (direct vs transitive), then sub-grouped by ecosystem:

<?xml version="1.0" encoding="UTF-8"?>
<opml version="2.0">
  <head>
    <title>SBOM Play — org / repo</title>
    <dateCreated>...</dateCreated>
    <ownerName>...</ownerName>
    <docs>http://opml.org/spec2.opml</docs>
  </head>
  <body>
    <outline text="Direct dependencies">
      <outline text="npm">
        <outline type="rss" xmlUrl="..." htmlUrl="..." title="..." text="..." category="..."/>
      </outline>
    </outline>
    <outline text="Transitive dependencies"> ... </outline>
  </body>
</opml>

Dedupe is performed by xmlUrl; when the same URL backs both a direct and a transitive dep, the direct entry wins. Uncovered packages are omitted entirely from the OPML; the page surfaces a skipped count so the export size is auditable.

Limitations:
  • Uncovered packages are silently omitted from the exported OPML — check the page table for the actual coverage rate.
  • Packagist requires a parseable vendor/name pair; packages with malformed PURL components fall through to the GitHub fallback.
  • The GitHub Releases atom feed only emits when the upstream maintainer publishes a GitHub Release object — tag-only repositories produce no entries.
