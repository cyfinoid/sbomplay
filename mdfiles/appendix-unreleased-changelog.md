### Added
- **About page now documents the recently added detection and scoring methodologies** (`about.html`): eight new explainer cards have been added alongside the existing EOX and Dependency Confusion sections, so users no longer have to read the codebase to understand what each new feature does. The new sections cover (1) **Package Lifecycle Status Detection (Official Signals)** — how npm `versions[ver].deprecated`, PyPI `info.status`, NuGet catalog deprecation, Cargo `version.yanked`, and the GitHub `archived` fallback feed `deprecated` / `yanked` / `archived` / `quarantined` statuses, with severity mappings into Findings and the replacement-hint extraction patterns; (3) **Malicious Package Detection Methodology** — the OSV/OpenSSF source, the four classifier rules (`kind === 'malware'`, `MAL-` prefix, `malicious-packages-origins`, `database_specific.malicious`), the `introduced: 0` trivial-range false-positive mitigation, and why malware is excluded from headline vulnerability counters; ### Fixed
- **Restored transitive dependency attribution everywhere — `licenses.html` reach sub-lines no longer all read `0 transitive`, the Insights depth chart no longer collapses every package into "direct" or "Unknown", and `deps.html` / `vuln.html` / `authors.html` / `findings.html` per-repo direct-vs-transitive counts agree with the v0.0.8 behaviour users expected** (`js/sbom-processor.js`, `js/storage-manager.js`): commits `b50fa97` ("table overflow fixed") and `efb35de` ("insights better alignment") landed a destructive post-pass in `SBOMProcessor.computeDependencyTrees` (and a mirror self-heal in `StorageManager._recomputeDirectAndTransitive`) that wiped every dep's `directIn` / `transitiveIn` / `repositories` and rebuilt them purely from `repo.dependencies` / `repo.directDependencies` (SBOM truth only). The intent was to fix a real ~87% cross-repo bleed bug in the resolver's parent-trace fallbacks, but the cure was worse than the disease: GitHub's dependency-graph SPDX SBOMs typically emit `DEPENDS_ON main → X` for every package, so every SBOM-listed dep landed in `repo.directDependencies` and the post-pass left no transitive candidates — the registry-built dep tree (npm / PyPI / Maven / Cargo / …) that v0.0.8 used to surface real transitive depth was thrown out. Net user impact: every reach sub-line on `licenses.html` (including the new Copyleft / Unknown / Total / Proprietary / Custom / Unlicensed cards) read `N direct / 0 transitive`, the Insights depth chart bucketed everything into "direct" or "Unknown", and the deps/vuln/authors per-repo direct-vs-transitive splits all collapsed. The fix replaces the destructive post-pass with a **two-pass attribution**: (1) **SBOM truth** — same as before, walk `repo.dependencies` and split on `repo.directDependencies`; (2) **per-repo BFS through `dep.children`** — for each repo, BFS from its SBOM-direct seed set through the registry-built resolver tree edges, attributing every reached dep to the repo as transitive (unless already direct in that repo via Pass 1). Pass 2 is structurally bleed-free: BFS only ever adds (dep, R) when `dep` is reachable from R's own direct deps via registry edges, so the original cross-ecosystem leak the `reposWithDirectDeps` fallback caused cannot recur. Resolver-discovered transitives that no SBOM listed (registry-fetched only — e.g. transitive `tar-stream` brought in by an `npm` direct dep but absent from the GitHub flat SBOM) finally land in the right repos instead of being orphaned. The inline parent-trace pass that used to live inside the per-ecosystem resolver loop (`js/sbom-processor.js` ~line 870-930) was deleted entirely — only the `dep.depth` / `dep.parents` / `dep.children` registry-tree-shape assignments remain, and all repo attribution is centralised in the post-pass for symmetry with `_recomputeDirectAndTransitive`. `StorageManager._recomputeDirectAndTransitive` is rewritten with the same two-pass algorithm so legacy stored analyses self-heal at load. A new `entry._directTransitiveHealVersion = 2` stamp replaces the previous boolean `_directTransitiveHealed` short-circuit so analyses that were already "healed" by the broken v1 pass (which wiped resolver attribution and stamped them `_directTransitiveHealed = true`) re-run through the new logic on next load — users do **not** need to re-scan to recover correct transitive numbers. The legacy boolean is still set to `true` for any code paths that may still read it.
- **`licenses.html` Unknown card headline (30) and click-through list (83) finally agree** (`js/view-manager.js` `generateLicenseTypesTableData`): the Unknown card's headline counter (`_computeLicenseReachCounts`) routed deps with `licenseInfo.license` of `null` / empty / `'NOASSERTION'` into the **unlicensed** bucket regardless of `licenseInfo.category`, but the click-through table treated `licenseInfo.category === 'unknown'` as the only filter criterion and let through every dep `LicenseProcessor.parseLicense` had classified as `'unknown'` (which is what it returns when there's nothing to classify). ### Fixed
- **`repos.html?repo=<owner/name>` URL parameter now actually filters the table** (`js/repos-page.js`): the page used to read only `?org` and `?search` — `?repo=` was silently ignored, so insights-page links like `repos.html?repo=Groww-OSS%2Freact-native-webview` rendered the full org list with no filter applied. - **Repo attribution on every dep was bleeding cross-repo, polluting the Insights "Dependency depth" chart, the Deps page, the Vulnerabilities page, and any other view that aggregates deps per repo** (`js/sbom-processor.js`, `js/storage-manager.js`, `js/insights-page.js`, plus matching cache-buster bumps on every HTML page that loads `sbom-processor.js` or `storage-manager.js`): the per-ecosystem dependency-tree resolver in `SBOMProcessor.computeDependencyTrees` runs each ecosystem in parallel and uses two over-broad fallbacks when wiring resolved transitives back to repos — (1) when a parent isn't yet in `this.dependencies` because its own ecosystem batch hasn't completed, the dep falls back to `reposWithDirectDeps` (every repo with any direct dep in the ecosystem); (2) when the dep ends up with zero repos via parent-trace, it inherits its first parent's `repositories` wholesale. Both fallbacks bled transitives across unrelated repos in the same ecosystem. The resolver's `dep.parents` / `dep.children` / `dep.depth` (registry-tree shape, used by vuln/license enrichment) are intentionally left intact. Resolver-discovered transitives that no SBOM listed (registry-only orphans) end up with empty `dep.repositories` and correctly stop appearing in any per-repo aggregation while remaining in `allDependencies` so enrichment data on them isn't lost.
